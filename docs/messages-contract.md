# messages contract

## goal
Define one shared messaging contract for web and mobile so both clients use the same database fields, rpc entry points, realtime topics, and read-state semantics.

## schema

### conversations
- `id bigint`
- `club_id bigint`
- `title text null`
- `is_group boolean`
- `created_by text`
- `created_at timestamptz`
- `updated_at timestamptz`
- `direct_key text null`

`direct_key` is set only for one-to-one direct messages. It is unique per club and per pair of users so the same DM always resolves to the same conversation.

### conversation_participants
- `id bigint`
- `conversation_id bigint`
- `user_id text`
- `joined_at timestamptz`
- `last_read_at timestamptz null`
- `last_read_message_id bigint null`

`last_read_message_id` is the source of truth for unread state. `last_read_at` remains for backward compatibility and analytics-friendly timestamps.

### messages
- `id bigint`
- `conversation_id bigint`
- `sender_id text`
- `body text`
- `created_at timestamptz`
- `updated_at timestamptz`
- `client_id uuid null`
- optional later: `edited_at timestamptz null`
- optional later: `deleted_at timestamptz null`

`client_id` is generated by the sending client and used for idempotency plus optimistic-send reconciliation.

## rpc contract

### `get_or_create_direct_conversation(club_id bigint, other_user_id text)`
- validates that the caller and target user are active members of the same club
- returns the existing one-to-one conversation id if it already exists
- otherwise creates the conversation and both participant rows atomically

response:

```json
{
  "conversation_id": 42
}
```

### `list_conversations(club_id bigint, cursor timestamptz default null, page_size integer default 20)`
- returns conversation summaries for the authenticated user in the selected club
- results are ordered by `updated_at desc`
- when `cursor` is present, only rows with `updated_at < cursor` are returned

### `list_messages(conversation_id bigint, cursor bigint default null, page_size integer default 30)`
- validates the caller is a conversation participant
- returns older messages when `cursor` is present using `message.id < cursor`
- response rows are sorted oldest-first within each page

response item:

### `send_message(conversation_id bigint, client_id uuid, body text)`
- validates the caller is a conversation participant
- trims and rejects empty bodies
- if the same sender retries with the same `client_id`, the existing row is returned
- updates the conversation timestamp and returns the canonical message payload

response:

```json
{
  "id": 91,
  "conversation_id": 42,
  "sender_id": "user_999",
  "body": "see you at practice",
  "created_at": "2026-03-12T22:05:00.000Z",
  "updated_at": "2026-03-12T22:05:00.000Z",
  "client_id": "0d71f2a6-ec35-4519-9064-6d4f66c0fef0",
  "sender": {
    "id": "user_999",
    "first_name": "Mia",
    "last_name": "Wong",
    "username": "mia",
    "avatar_url": null
  }
}
```

### `mark_conversation_read(conversation_id bigint, last_read_message_id bigint)`
- validates the caller is a conversation participant
- only moves the read pointer forward
- returns the effective stored read pointer

response:

```json
{
  "last_read_message_id": 91
}
```

## realtime contract

### topics
- `conversation:{id}:messages`
- `conversation:{id}:presence`
- `user:{userId}:conversations`

All database-triggered topics are private channels.

### events
- `message_created`
- `message_updated`
- `message_deleted`
- `conversation_updated`
- `typing_started`
- `typing_stopped`

### event payloads

Conversation messages topics broadcast database row changes. The web client fetches the canonical joined message payload after receiving `message_created` or `message_updated`.

Example `conversation_updated` payload on `user:{userId}:conversations`:

```json
{
  "conversation_id": 42,
  "club_id": 7
}
```

Example typing payload on `conversation:{id}:presence`:

```json
{
  "conversation_id": 42,
  "user_id": "user_999",
  "is_typing": true,
  "sent_at": "2026-03-12T22:05:00.000Z"
}
```

## behavior rules
- only active club members can start or participate in direct conversations
- plain message delivery uses unread state only, not the separate in-app notification feed
- unread is cleared only when the thread is focused and the latest visible message has been reached
- typing is ephemeral only and must never be stored in the database
- typing should auto-expire after 2 to 3 seconds of inactivity
- ids are normalized as JavaScript numbers on web clients

## Verification (cross-platform smoke check)

After applying the migration `20260312220240_harden_messages_contract.sql` to your Supabase project:

1. Start the same DM from web and mobile; both must resolve to one shared conversation.
2. Send from web; mobile thread must update in realtime.
3. Send from mobile; web thread must update in realtime.
4. With the thread closed, confirm unread increments in the list and nav badge only.
5. Open the thread and confirm unread clears only after the latest message is visible.
6. Type without sending; typing must appear and expire after 2–3 seconds.
7. Retry a send with the same `client_id`; no duplicate message must be created.

Apply the migration via Supabase CLI (`supabase db push`) or Dashboard SQL editor before running these checks.
